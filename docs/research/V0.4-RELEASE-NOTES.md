## 1. v0.4 概要 — "Preview Revolution + Intelligence + Power & Discovery"

`v0.4` シリーズは、`v0.3` までで固まった「strip-first / shared carousel / folder warehouse / snippet authoring」の土台に対して、**ultrathink プラン** から派生した次の 3 軸を一気に重ねる計画である。

| 軸 | 目的 | 代表機能 |
|---|---|---|
| **Preview Revolution** | 「貼る前に中身が見える」をマウスもキーボードも問わず実現する | `QuickLookPreview` (Space) / `ClipPreviewView` / `HoverPreviewController` (600ms ホバー) / `LinkMetadataFetcher` (URL リッチカード) |
| **Intelligence** | クリップを「ただ並んだ履歴」から「理解された情報」に昇格させる | `AIEngine` (Foundation Models 統合 + ヒューリスティクス) / `AIActionMenu` / `AutoCategorizer` / `SmartFolder` |
| **Power & Discovery** | パワーユーザ向けキー操作と、新規ユーザのオンボーディングを両立 | `TemplateFieldDialog` (`[[name]]`) / `SnippetEngine` 動的修飾子 / `OnboardingView` / `HelpOverlayView` (`⌘?`) / `HotkeyRecorder` / `InsightsDashboard` / `ImportExport` / `StackPill` |

リリースは 2 段で投入した:

1. **v0.4.0 (2026-06-13 15:21 JST)** — 15 体並列エージェントによる「ファイル投入」。
2. **v0.4.1 (2026-06-13 15:39 JST)** — 投入したファイル群を **配線** し、実動作するように繋ぎ込む。

`v0.4.0` 単体ではまだ「並んでるだけのファイル」が一部あり、`v0.4.1` で `IntegrationHubs.swift` を新設してハブ化することで初めて全体が動く構成になっている。

### 2.4 ビルド事故と手動修正

並列投下後のビルドで以下の階級の問題が出た。順次手動修正:

- **actor 隔離**: `QLPreviewPanelDataSource` が nonisolated で呼ばれる前提で main actor 隔離の `ClipStore` を直接掴めない → `PreviewDataSource` 側で snapshot 化して保持。
- **`public` / `internal` の不一致**: いくつかの新規型が `internal` で書かれていて、別ファイルから呼べないアクセス制御エラー。
- **`enum` 重複**: `AIEngine.Action` 系の case と UI 側 (`AIActionMenu`) が二重定義しかけた箇所を extension で吸収。

## 4. 設計判断ログ

### 4.1 なぜ `[[name]]` を選んだか

mail-merge プレースホルダの記法として `{{name}}` `${name}` `<<name>>` `[[name]]` を検討した結果、**`[[name]]`** を採用。理由は順に:

1. **`{{...}}` は既に動的修飾子チェーンに使用**: `SnippetEngine` は `{{var | uppercase | trim}}` のテンプレ展開を持っており、`{{name}}` は「定義済みのトークンを評価する」セマンティクスに固定済み。mail-merge の「ユーザに対話的に値を尋ねる」セマンティクスは別記号で表す方が衝突しない。
   - `SnippetEngine.swift` の実装でも、まず `{{...|...}}` を先に貪欲マッチして消費してから `[[...]]` をパースする順序になっている（コメント: *"...consumed first and don't get mis-parsed."*）。
2. **`${name}` はシェル/JS テンプレ文字列と被る**: クリップに貼られる素材自体に `${...}` を含むケースが多すぎる。
3. **`<<name>>`** は Markdown / HTML との視覚的衝突が多い。
4. **`[[name]]`** は Roam Research や Obsidian の wikilink で既に "プレースホルダ感" を持つ記法として広く知られている。ユーザ学習コストが低い。

### 4.2 Foundation Models のフォールバック戦略

`AIEngine.swift` は **二段戦略** で構築:

```swift
static var isFoundationModelsAvailable: Bool {
    #if canImport(FoundationModels)
    if #available(macOS 26.0, *) {
        return SystemLanguageModel.default.isAvailable
    }
    #endif
    return false
}
```

- **第 1 経路**: macOS 26+ かつ Apple Intelligence が利用可能なら `LanguageModelSession()` で Foundation Models を直接叩く。
- **第 2 経路**: それ以外（macOS 25 以下 / ユーザがオフ / モデル未 DL）はヒューリスティック実装で同じ `Action` enum を満たす。

この設計の狙い:

| 観点 | 採用理由 |
|---|---|
| **ビルドターゲット** | Xcode の `#if canImport(FoundationModels)` で SDK が無い環境（Xcode 15 以下）でもコンパイル可能。 |
| **実行時可用性** | `#available` + `SystemLanguageModel.default.isAvailable` の二段チェックで、ユーザが Apple Intelligence をオフにしているケースを取りこぼさない。 |
| **UX 一貫性** | フォールバック側でも「結果は新クリップに保存」フローを共通にしているので、ユーザから見て「AI が無効でもとりあえず何か返る」状態を維持。 |
| **ネット非依存** | 第二経路に外部 API を一切使わない。プライバシーノイズになる前に切り離している。 |

### 4.3 QuickLook を `QLPreviewPanel` ベースにした理由

`Space` で大画面プレビューする選択肢は 3 つあった:

1. SwiftUI で自作 modal sheet
2. `NSPanel` をフルスクリーン化して `ClipPreviewView` を埋める
3. `QLPreviewPanel.shared()` を直接掴む

採用は **3**。

- **OS 標準の Quick Look** に乗ることで、画像/PDF/コード/Office 系/動画 まで OS が知ってるフォーマット全部を **ノーコードで** カバーできる。
- 制御権が OS に渡るので、Pasty 側パネル (`StripPanel` / `SpotlightPanel`) は閉じる必要がなく、**Space を離した瞬間に元の絞り込み状態に戻れる**。
- ただし `QLPreviewPanelDataSource` / `QLPreviewPanelDelegate` は `nonisolated` で呼ばれるため、`@MainActor` の `ClipStore` を直接掴めない。これは `PreviewDataSource` 側に **snapshot 配列を保持** する形で解決している (`QuickLookPreview.swift` 内部の `WrappedItem`)。
- テキスト/コード/Markdown は QuickLook が直接表示できないので、**一時ファイルにシンタックスハイライト付き HTML を書き出して** `QLPreviewItem.previewItemURL` に渡す方式。

## 6. 既知の制約 / 次の P8 候補

### 6.1 `paste_count` カラムが未実装

`InsightsDashboard.swift` は集計時に `paste_count` カラムの存在を **runtime で動的に検出** している:

```swift
// paste_count カラムの有無を検出
snap.hasPasteCountColumn = columns.contains { $0.name == "paste_count" || $0.name == "pasteCount" }
...
let pcColumn = columns.contains { $0.name == "paste_count" } ? "paste_count" : "pasteCount"
```

つまり **現時点で実テーブルに `paste_count` カラムは無い**。`PasteHistory.shared.record` も in-memory のみで永続化していない。

> 永続化は v0.5 で `paste_events` テーブルとして再設計予定。 *(IntegrationHubs.swift コメントより)*

設計案:

| カラム | 型 | 用途 |
|---|---|---|
| `clip_id` | TEXT (FK) | 対応するクリップ |
| `pasted_at` | INTEGER (epoch ms) | 貼付時刻 |
| `dest_bundle_id` | TEXT NULL | 貼付先アプリ識別 |
| `transformer` | TEXT NULL | snippet / AI action / mail-merge いずれが噛んだか |

これにより Insights ダッシュボードの "best time of day" / "by-app heatmap" が実値で動く。

### 6.2 フォルダ階層 (`parentId`) 未対応

`SmartFolder` および「フォルダ倉庫」は **フラット構造**。`parentId` を持たない。サブフォルダ / ネスト構造は v0.5 候補。

### 6.3 iCloud 同期未実装

`ImportExport.swift` で JSON Export/Import は提供しているが、**iCloud Drive / CloudKit による自動同期は無し**。

- 画像 blob は base64 で同梱しているため、Export 単発で 100 MB+ になり得る。
- インクリメンタル同期、conflict 解決、デバイス間 dedup は v0.5 設計フェーズで決定する。

### 6.4 MCP server stub

外部から Pasty クリップを叩く MCP サーバ層は **未実装**。Claude / 他エージェントから「最近の n 件を読む」「クリップを書き込む」を提供するなら次の P8 タスク。

### 6.5 その他

- `HoverPreviewController` の 600ms 遅延はハードコード。`SettingsStore` 露出は未。
- `LinkMetadataFetcher` のディスクキャッシュ TTL は無期限。
- `AIActionCoordinator` 結果クリップに `kind = .ai` のようなフラグを立てておらず、通常テキストクリップとして混在する。
